# Plan B: Lightweight Structured Feedback via Push Hooks and Opt-In Surveys

## Goal

A lighter-weight alternative to Plan A that minimises workflow complexity. Instead of collecting all agent report files on every push, this plan focuses on:

1. **Structured metadata tags** appended automatically by the CI pipeline on each push.
2. **A short opt-in survey** that contributors answer once when they complete a project milestone.
3. **Aggregated periodic summaries** rather than per-push dispatches.

The result is less noise in the template repo and more signal in the feedback that does arrive.

---

## Core Ideas

### Idea 1 — Structured Metadata Tags

Each push appends a small structured block to `feedback/push-log.jsonl` (newline-delimited JSON). The block is generated by the CI workflow and contains only metadata — no code, no report content:

```json
{"date":"2025-06-01T12:00:00Z","repo":"owner/myproject","sha":"abc123","agents_run":["planner","worker","reviewer"],"tests_passed":true,"issues_found":2,"lessons_count":5}
```

**Benefits:**
- No large payloads — each entry is one line.
- Easily aggregated with `jq` or any log tool.
- No AI review needed on every push — the template can aggregate weekly.

**How it works:**
- The push CI workflow runs a small script that reads counts from the existing report files (no content, just line counts or section counts).
- The script appends a JSON line to `feedback/push-log.jsonl`.
- On a schedule (e.g. weekly), a separate workflow in the template repo reads all incoming `push-log.jsonl` entries and sends the batch to Claude for summary.

---

### Idea 2 — Milestone Survey (FEEDBACK_SURVEY.md)

When a user completes a milestone (e.g. ships their first feature using the template), they fill in `feedback/FEEDBACK_SURVEY.md` and push. The file contains structured questions:

```markdown
# Feedback Survey

**Milestone:** First Feature Shipped

**Q1 — What part of the template helped most?**
The agent pipeline made scaffolding fast.

**Q2 — What was most confusing or frustrating?**
The Librarian context brief was not clear on when to use it.

**Q3 — What one thing would you change?**
Add a QUICK_START.md with a 5-minute example.

**Q4 — Overall rating (1–5):**
4
```

The push trigger detects that the survey has meaningful content — defined as: at least one answer line that does not start with `<!--`, is not blank, and is not the literal placeholder text `_your answer here_` — and sends it via issue-mode to the template repo. Claude reviews it and creates a PR.

```bash
# Detect meaningful content: count non-blank, non-comment, non-placeholder lines after the header
meaningful=$(grep -v '^\s*$' feedback/FEEDBACK_SURVEY.md \
  | grep -v '^\s*<!--' \
  | grep -v '^#' \
  | grep -v '_your answer here_' \
  | wc -l)
if [ "$meaningful" -lt 1 ]; then
  echo "Survey is blank — skipping send."
  exit 0
fi
```

After sending, the workflow resets the survey file to its blank template.

### Idea 3 — Weekly Aggregation Workflow

Instead of the template repo processing every push, it runs a scheduled workflow on Monday mornings:

```yaml
on:
  schedule:
    - cron: '0 8 * * MON'   # Every Monday at 08:00 UTC
```

The workflow:
1. Reads all `.jsonl` entries in `feedback/` committed since the last run.
2. Reads all `*_claude_review.md` files created since the last run.
3. Passes the aggregated data to Claude with the prompt: "Summarise the patterns you see across these projects and propose template improvements."
4. Opens a single weekly improvement PR.

**Benefits:**
- One PR per week instead of one per push.
- Claude can spot cross-project patterns that per-push reviews miss.
- Lower API cost.

---

### Idea 4 — Agent Self-Reporting Hook

Each agent (Worker, Reviewer, Retrospective, etc.) already appends to `docs/RETROSPECTIVE_REPORT.md` and `.ai/lessons.md`. Add a lightweight hook to each agent's instructions:

> At the end of your report, append a `## Template Feedback` section with any observations about the template itself — confusing instructions, missing patterns, or rules that conflicted.

This section is parsed by the feedback workflow and extracted as a dedicated "agent notes" payload, separate from the project-level retrospective. It gives the template direct signal from the AI agents about what is hard to follow.

## Comparison with Plan A

| Dimension | Plan A | Plan B |
| --- | --- | --- |
| Feedback frequency | Every push (with guard) | Opt-in survey + weekly batch |
| Payload size | Full report files (up to 500 lines each) | Metadata line + survey text |
| Template repo noise | One PR per meaningful push | One PR per week |
| User effort | Zero (fully automatic) | One survey fill per milestone |
| Agent self-reporting | Passive (reads existing files) | Active (agent appends a feedback section) |
| Best for | Projects with active agent sessions | Projects with infrequent pushes or solo devs |

## Privacy Considerations

Same as Plan A:

- Metadata only in `push-log.jsonl` — no source code.
- Survey is fully opt-in.
- All dispatch happens only when `TEMPLATE_FEEDBACK_ENABLED=true`.

---

## Recommended Combination

Use **Plan A** as the default for agent-generated report files (they change on their own, no user action needed). Use **Plan B, Idea 2** (the milestone survey) on top of Plan A to capture human voice feedback at natural checkpoints. Use **Plan B, Idea 3** (weekly aggregation) in the template repo to reduce PR noise.

This gives the template:
- Continuous automatic signal from AI agents.
- Periodic high-quality signal from humans.
- A single weekly PR with cross-project insights.
